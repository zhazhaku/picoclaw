# Reef v2.0 — 详细规格说明

## 1. 持久化任务队列

### Requirement: TaskStore 接口
系统 SHALL 定义 `TaskStore` 接口，支持任务的 CRUD 和查询操作。

#### Scenario: TaskStore 接口定义
- GIVEN `pkg/reef/server/store/` 包
- WHEN 定义 `TaskStore` 接口
- THEN 接口包含以下方法：
  - `SaveTask(task *reef.Task) error`
  - `GetTask(id string) (*reef.Task, error)`
  - `UpdateTask(task *reef.Task) error`
  - `DeleteTask(id string) error`
  - `ListTasks(filter TaskFilter) ([]*reef.Task, error)`
  - `SaveAttempt(taskID string, attempt reef.AttemptRecord) error`
  - `GetAttempts(taskID string) ([]reef.AttemptRecord, error)`
  - `Close() error`

### Requirement: SQLite 持久化后端
系统 SHALL 实现 SQLite WAL 模式的持久化存储。

#### Scenario: SQLite 存储任务
- GIVEN Server 配置 `store_type = "sqlite"`，`store_path = "/data/reef.db"`
- WHEN 通过 Admin API 提交一个任务
- THEN 任务被写入 SQLite 数据库的 `tasks` 表
- AND 任务的尝试记录写入 `task_attempts` 表

#### Scenario: Server 重启恢复任务
- GIVEN Server 已提交 5 个任务（2 个 Running，3 个 Queued）
- WHEN Server 重启
- THEN 从 SQLite 加载所有非终态任务
- AND Running 任务标记为 Queued（需要重新调度）
- AND Queued 任务保持 Queued 状态
- AND 已完成/已取消的任务不加载到内存

#### Scenario: WAL 模式并发读写
- GIVEN SQLite 存储启用 WAL 模式
- WHEN 多个 goroutine 同时读写任务
- THEN 不发生死锁或数据损坏
- AND 读操作不阻塞写操作

#### Scenario: 内存存储向后兼容
- GIVEN Server 配置 `store_type = "memory"`（或未配置）
- WHEN Server 启动
- THEN 使用纯内存存储，行为与 v1.1 完全一致
- AND 不创建 SQLite 文件

### Requirement: 配置驱动存储选择
系统 SHALL 通过配置选择存储后端。

#### Scenario: 配置 store_type
- GIVEN `config.json` 中 `channels.swarm.store_type = "sqlite"`
- AND `channels.swarm.store_path = "/data/reef.db"`
- WHEN Server 启动
- THEN 使用 SQLite 存储
- AND 数据库文件创建在指定路径

#### Scenario: 默认使用内存存储
- GIVEN 未配置 `store_type`
- WHEN Server 启动
- THEN 使用内存存储

## 2. 性能基线测试

### Requirement: 基准测试框架
系统 SHALL 提供可复现的性能基准测试框架。

#### Scenario: 任务提交吞吐测试
- GIVEN 一个 Reef Server 和 N 个连接的 Client
- WHEN 以并发方式提交 M 个任务
- THEN 测量并报告：总耗时、吞吐（tasks/sec）、p50/p95/p99 提交延迟

#### Scenario: 调度延迟测试
- GIVEN 一个 Reef Server 和 1 个连接的 Client
- WHEN 提交一个任务
- THEN 测量从提交到 Client 收到 `task_dispatch` 的延迟

#### Scenario: WebSocket 消息吞吐测试
- GIVEN 一个 Reef Server 和 N 个连接的 Client
- WHEN 所有 Client 同时发送心跳消息
- THEN 测量 Server 处理心跳的吞吐（msg/sec）

#### Scenario: 并发连接压力测试
- GIVEN 一个 Reef Server
- WHEN 同时建立 10/50/100 个 WebSocket 连接
- THEN 所有连接成功建立
- AND 注册消息全部被正确处理
- AND 测量连接建立的 p99 延迟

#### Scenario: JSON 报告输出
- GIVEN 任何基准测试完成
- WHEN 测试结束
- THEN 输出 JSON 格式报告到 `test/perf/results/`
- AND 报告包含：test_name、timestamp、concurrency、total_tasks、duration_ms、throughput、latency_percentiles、error_rate

### Requirement: 性能回归检测
系统 SHALL 支持性能基准对比。

#### Scenario: 基线对比
- GIVEN 已有基线报告 `results/baseline.json`
- WHEN 运行新的基准测试
- THEN 自动对比关键指标
- AND 如果 p99 延迟增加 > 20% 或吞吐下降 > 15%，标记为回归

## 3. Web UI 仪表盘

### Requirement: 嵌入式 Web UI
系统 SHALL 提供嵌入式 Web UI，通过 Admin 端口提供服务。

#### Scenario: 访问 Web UI
- GIVEN Server 启动且 Admin 端口为 `:8081`
- WHEN 浏览器访问 `http://server:8081/`
- THEN 返回 Web UI 页面
- AND 页面加载成功，无外部 CDN 依赖

#### Scenario: 概览页面
- GIVEN Web UI 已加载
- WHEN 用户查看概览页面
- THEN 显示：Server 版本、运行时间、连接客户端数、队列深度、任务统计（成功/失败/运行中）

#### Scenario: 任务列表页面
- GIVEN Web UI 已加载
- WHEN 用户查看任务列表
- THEN 显示所有任务的 ID、状态、角色、创建时间、分配客户端
- AND 支持按状态/角色筛选
- AND 支持分页（每页 50 条）

#### Scenario: 客户端拓扑页面
- GIVEN Web UI 已加载且有 3 个客户端连接
- WHEN 用户查看客户端页面
- THEN 显示每个客户端的 ID、角色、技能、状态、负载、最后心跳时间

#### Scenario: 实时更新
- GIVEN Web UI 已打开
- WHEN 一个新任务被提交或状态变更
- THEN UI 自动更新（通过 WebSocket 推送或 SSE）
- AND 无需手动刷新

#### Scenario: 提交任务
- GIVEN Web UI 已打开
- WHEN 用户填写任务表单（instruction、role、skills）并提交
- THEN 任务通过 Admin API 提交
- AND 任务出现在任务列表中

## 4. TLS 原生支持

### Requirement: Server TLS 配置
系统 SHALL 支持为 WebSocket 和 Admin 端口配置 TLS。

#### Scenario: 启用 TLS
- GIVEN Server 配置了 `tls.cert_file` 和 `tls.key_file`
- WHEN Server 启动
- THEN WebSocket 监听 `wss://` 协议
- AND Admin API 监听 `https://` 协议

#### Scenario: Client 连接 wss://
- GIVEN Server 启用了 TLS
- AND Client 配置了 `server_url = "wss://server:8080"`
- WHEN Client 连接
- THEN WebSocket 连接通过 TLS 建立
- AND 任务调度正常工作

#### Scenario: 自定义 CA 证书
- GIVEN Server 使用自签名证书
- AND Client 配置了 `tls.ca_file = "/path/to/ca.pem"`
- WHEN Client 连接
- THEN TLS 握手成功（使用自定义 CA 验证）

#### Scenario: TLS 未配置时行为不变
- GIVEN 未配置 TLS
- WHEN Server 启动
- THEN WebSocket 监听 `ws://`（明文）
- AND Admin API 监听 `http://`（明文）
- AND 行为与 v1.1 完全一致

## 5. 多通道告警通知

### Requirement: Notifier 接口
系统 SHALL 定义 `Notifier` 接口支持多种通知渠道。

#### Scenario: Notifier 接口
- GIVEN `pkg/reef/server/notify/` 包
- WHEN 定义 `Notifier` 接口
- THEN 接口包含 `Notify(ctx context.Context, alert Alert) error` 方法
- AND `Alert` 结构体包含 event、task_id、status、message、timestamp

### Requirement: Slack 通知
系统 SHALL 支持通过 Slack Incoming Webhook 发送告警。

#### Scenario: Slack 告警
- GIVEN 配置了 `notifications = [{type: "slack", webhook_url: "https://hooks.slack.com/..."}]`
- WHEN 任务升级到 Admin
- THEN 向 Slack 发送消息
- AND 消息包含任务 ID、状态、错误信息

### Requirement: 邮件通知
系统 SHALL 支持通过 SMTP 发送告警邮件。

#### Scenario: SMTP 告警
- GIVEN 配置了 SMTP 通知（host、port、from、to、credentials）
- WHEN 任务升级到 Admin
- THEN 发送告警邮件到配置的收件人
- AND 邮件主题包含 `[Reef Alert]` 前缀

### Requirement: 飞书通知
系统 SHALL 支持通过飞书 Webhook 发送告警。

#### Scenario: 飞书告警
- GIVEN 配置了飞书 Webhook URL
- WHEN 任务升级到 Admin
- THEN 向飞书发送富文本卡片消息

### Requirement: 企业微信通知
系统 SHALL 支持通过企业微信 Webhook 发送告警。

#### Scenario: 企业微信告警
- GIVEN 配置了企业微信 Webhook URL
- WHEN 任务升级到 Admin
- THEN 向企业微信发送 Markdown 消息

### Requirement: 通知扇出
系统 SHALL 支持同时配置多个通知渠道。

#### Scenario: 多渠道同时通知
- GIVEN 配置了 Slack + 邮件 + 飞书三个通知渠道
- WHEN 任务升级
- THEN 三个渠道都收到通知
- AND 任一渠道失败不影响其他渠道

## 6. 多 Server 联邦（探索性）

### Requirement: Raft 共识
系统 SHALL 支持多 Server 节点通过 Raft 协议达成共识。

#### Scenario: Leader 选举
- GIVEN 3 个 Server 节点组成集群
- WHEN 集群启动
- THEN 选举出一个 Leader
- AND 其他节点为 Follower

#### Scenario: Leader 故障转移
- GIVEN 3 节点集群，当前 Leader 正常
- WHEN Leader 节点宕机
- THEN 剩余节点在 5 秒内选举新 Leader
- AND 新 Leader 接管任务调度

#### Scenario: 任务路由到 Leader
- GIVEN 集群已选举 Leader
- WHEN Client 连接到任意 Server 节点
- THEN 任务请求被路由到 Leader 处理
- AND 任务结果返回给 Client

**注意：联邦特性为探索性，复杂度高，可能推迟到 v2.1 或 v3。**
